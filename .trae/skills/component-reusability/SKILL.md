# 前端组件复用性优化指南

## 触发条件（When）

当用户需要处理组件复用性相关问题时调用，包括：
- 创建新的UI组件
- 发现组件代码重复
- 重构现有组件
- 设计组件架构
- 用户提到"组件复用性不足"

## 执行步骤（How）

### 1. 遵循核心原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 每个组件只负责一个功能点 |
| 可组合性 | 组件能够灵活组合，支持多种场景 |
| Props设计 | 最小化Props，提供合理默认值 |
| 插槽设计 | 使用插槽提供灵活性 |

### 2. 设计通用组件

按以下优先级设计：

| 组件类型 | 优先级 | 示例 |
|----------|--------|------|
| 基础组件 | ⭐⭐⭐⭐ | Button、Input、Select |
| 布局组件 | ⭐⭐⭐⭐ | Modal、Card、Table |
| 表单组件 | ⭐⭐⭐ | Form、FormItem |
| 复合组件 | ⭐⭐⭐ | Tag、Badge |

### 3. 应用优化策略

- 提取公共逻辑到 Composables
- 使用 HOC 添加通用功能
- 使用 Renderless Components 分离逻辑和UI
- 使用 Provide/Inject 跨层级通信

## 输出结果（What）

可复用、可维护的组件方案，包含：
- 清晰的组件职责划分
- 合理的 Props 和插槽设计
- 完整的类型定义
- 可扩展的组件架构

## 组件复用性检查清单

### 设计阶段
- [ ] 组件职责是否单一？
- [ ] Props设计是否合理？是否有默认值？
- [ ] 是否提供了足够的插槽？
- [ ] 是否支持多种使用场景？

### 实现阶段
- [ ] 是否使用了TypeScript类型定义？
- [ ] 是否提供了完整的事件回调？
- [ ] 样式是否可定制？
- [ ] 是否支持国际化？

### 测试阶段
- [ ] 是否编写了单元测试？
- [ ] 是否测试了各种Props组合？
- [ ] 是否测试了边界情况？

## 最佳实践

1. **设计优先**：先设计组件API，再实现功能
2. **渐进增强**：从简单到复杂，逐步完善
3. **文档先行**：编写清晰的文档和示例
4. **测试保障**：编写充分的单元测试
5. **持续重构**：根据使用反馈持续优化
6. **性能意识**：关注组件性能，避免不必要的渲染
7. **可访问性**：考虑无障碍访问，支持键盘操作
8. **国际化**：支持多语言，使用i18n
